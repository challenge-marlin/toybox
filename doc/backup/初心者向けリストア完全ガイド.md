# TOYBOXバックアップ・リストア完全ガイド

## このガイドについて

このガイドでは、**データベース知識がない方でも**、TOYBOXのバックアップデータを復元できるよう、画面操作から順番に説明します。

**対象**：
- 初心者向けの分かりやすい手順
- データベース管理者向けの技術情報
- トラブルシューティング

**作成日**：2026年1月29日  
**最終更新**：2026年2月5日（v3.0）

### リストアとは？

「リストア」とは、バックアップしたデータを元に戻す作業のことです。

### リストアに必要なもの

TOYBOXのリストアには**2つのファイル**が必要です：

1. **データベースファイル**
   - ユーザー情報、投稿データ、カード情報など
   - 形式：`.dump`（カスタムフォーマット）
   - 例：`toybox_20260202_210001.dump`

2. **メディアファイル**（`.tar.gz`）
   - 画像、動画、ゲームファイルなど
   - 例：`media_volume_20260202_210001.tar.gz`

**⚠️ 重要**：必ず**同じ日時のファイル**を両方とも使ってください！

### データベースファイルの種類

| 形式 | 拡張子 | 特徴 | リストア方法 | 推奨度 |
|------|--------|------|-------------|--------|
| **カスタムフォーマット** | `.dump` | バイナリ形式、高速、**文字化けしない** | `pg_restore` | ⭐⭐⭐（推奨） |
| SQLテキスト | `.sql` / `.sql.gz` | テキスト形式（確認しやすい）だが文字化けリスクあり | `psql` | △（原則使わない） |

**📌 重要**：
- **`.dump`形式でバックアップし、`.dump`形式でリストア**します（推奨）
- `.sql.gz`形式は文字化けリスクがあるため、原則として使いません（古いバックアップがある場合のみ）

## 自動バックアップはどうやって作られている？（初心者向け）

### いつ取られる？

- **毎日 21:00** に自動バックアップが実行されます（cron等のスケジューラ）。

（技術者向けメモ）cronの例：

```bash
# 毎日21:00に実行（サーバー側）
# ※ /usr/bin/bash で実行（改行コード問題を回避）
0 21 * * * /usr/bin/bash /var/www/toybox/scripts/backup_nightly.sh
```

### 何が作られる？

自動バックアップでは、以下の2つが作られます：

1. **データベース（`.dump`）**
   - 保存先：`/backup/toybox/database/`
   - ファイル名例：`toybox_20260202_210001.dump`

2. **メディアボリューム（`.tar.gz`）**
   - 保存先：`/backup/toybox/volumes/`
   - ファイル名例：`media_volume_20260202_210001.tar.gz`

### なぜバックアップ中にWebを止めるの？

バックアップ中に投稿やアップロードが走ると、

- データベースとメディアの時刻がズレる
- 「DBにはURLがあるのにメディアが無い」など不整合が起きる

といった事故が起きやすくなります。

そのため、**バックアップ中は一時的にWeb（投稿処理）を停止**して、同じ時刻のDBとメディアをセットで取得します。

### 重要：同じ時刻の2ファイルをセットで使う

リストア時は必ず、**同じ時刻の2ファイル**を使います：

- `toybox_YYYYMMDD_HHMMSS.dump`
- `media_volume_YYYYMMDD_HHMMSS.tar.gz`

どちらか片方だけでは、画面表示が壊れたり、画像/動画が出なかったりします。

### いつ使う？

このガイドは以下のような場合に使います：
- **データ損失が発生した時**（誤って削除、障害など）
- 本番サーバーのデータを別のサーバーに移す
- 本番サーバーのデータを自分のPCで確認する

### ⚠️ 絶対に守ること

**事故が起きた後に本番サーバーでバックアップを取らないでください！**
- 既に欠損したデータがバックアップされてしまいます
- **必ず事故前に取得した既存のバックアップファイルを使います**
- バックアップは定期的に自動取得されているので、それを使用します

**リストア手順は必ず2つとも実行してください：**
- ✅ データベース（SQL）のリストア
- ✅ メディアボリュームのリストア
- どちらか片方だけでは不完全です！

### 文字化け問題について

リストア後に日本語が「?????」と表示される問題が発生することがあります。

**症状例**：
- カード名が「トラブルリキッド・セイバー」→「?????????????」
- ユーザー名が正しく表示されない

**原因**：
- **`.sql.gz`形式**：PowerShellパイプ経由でリストアすると文字化けする
- UTF-8エンコーディングの設定が不足
- データベースに保存される時点で文字化けしている

**✅ 解決策**：
- **`.dump`形式を使用する**（2026年1月30日より推奨）
  - バイナリ形式なので文字化けが起きない
  - 自動バックアップは既に`.dump`形式に変更済み
- `.sql.gz`形式を使う場合は、コンテナ内で直接リストア実行

**このガイドでは文字化けを防ぐ方法を含めて説明しています**
- 正しいリストア手順を踏めば、文字化けは発生しません
- もし文字化けが発生した場合は、[トラブルシューティング](#問題1-日本語がと表示される)を参照してください

## 準備するもの

### 必要なソフトウェア

1. **WinSCP**（ファイル転送ツール）
   - サーバーからファイルをダウンロードするために使います
   - 視覚的に操作できるので初心者にも分かりやすい
   - 公式サイト：https://winscp.net/eng/download.php

2. **PowerShell**（Windowsに最初から入っているツール）
   - サーバーへの接続とコマンド実行に使います
   - ローカルPCでの作業にも使います
   - Windowsの検索で「PowerShell」と入力すれば見つかります

3. **Docker Desktop**（ローカル検証の場合のみ）
   - ローカルPCでDockerを動かすために必要
   - インストール済みであることを確認してください
   - 公式サイト：https://www.docker.com/products/docker-desktop

4. **サーバーのログイン情報**
   - サーバーのIPアドレス（例：`123.456.789.0`）
   - ユーザー名（通常は `root`）
   - パスワード

5. **Cursor AI**（このツール！）
   - 分からないことがあれば、いつでも聞いてください
   - エラーメッセージをそのまま貼り付ければ、解決方法を教えてくれます

## パターン1: サーバー間のリストア

> **例：既存のバックアップから本番サーバーまたはステージングサーバーにデータを復元**

### ステップ1: 既存のバックアップファイルを確認

#### 1-1. PowerShellで本番サーバーに接続

1. **PowerShellを起動**
   - Windowsの検索で「PowerShell」と入力
   - 「Windows PowerShell」をクリック

2. **サーバーにSSH接続**
   - 以下のコマンドを入力してEnterキーを押します：
   ```powershell
   ssh root@本番サーバーのIPアドレス
   ```
   - 例：`ssh root@123.456.789.0`

3. **パスワードを入力**
   - 「password:」と表示されたら、サーバーのパスワードを入力
   - **注意**：入力中は画面に何も表示されませんが、正常です
   - 入力したらEnterキーを押します

4. **接続成功の確認**
   - プロンプトが `root@vm-XXXX:~#` のように変わればOK

**💡 ヒント**：
- 初回接続時に「Are you sure you want to continue connecting?」と聞かれたら `yes` と入力してEnter
- SSH接続がうまくいかない場合は、Cursor AIに「SSH接続できません」と相談してください

#### 1-2. 既存のバックアップファイルを確認

PowerShellに以下のコマンドを入力して、**事故前の最新バックアップ**を確認します：

```bash
# データベースバックアップを確認
ls -lh /backup/toybox/database/ | tail -5

# メディアバックアップを確認
ls -lh /backup/toybox/volumes/ | tail -5
```

**⚠️ 重要**：
- **事故が起きた日時より前**のバックアップファイルを選んでください
- 日付と時刻がファイル名に含まれています
- 例：`toybox_20260129_153045.dump`（2026年1月29日 15:30:45のバックアップ）

#### 1-3. バックアップファイル名をメモ

使用するバックアップファイル名をメモしてください。
例：
- データベース：`toybox_20260129_153045.dump`
- メディア：`media_volume_20260129_153045.tar.gz`

**💡 ヒント**：
- データベースとメディアは**同じ日時のファイル**を選ぶのが最適です
- どのファイルを使うか分からない場合は、Cursor AIに相談してください

### ステップ2: ステージングサーバーまたは本番サーバーにリストア

#### 2-1. PowerShellで本番サーバーから切断（必要な場合）

現在のPowerShellで本番サーバーに接続している場合は、以下を入力して切断します：

```bash
exit
```

#### 2-2. バックアップファイルを復元先サーバーにコピー

##### 方法A: WinSCPで転送（推奨・視覚的で分かりやすい）

**手順**：

1. **WinSCPで本番サーバーに接続**
   - ホスト名：本番サーバーのIPアドレス
   - ユーザー名：`root`
   - パスワード：本番サーバーのパスワード

2. **右側（サーバー側）で以下のフォルダを開く**：
   ```
   /backup/toybox/database/
   ```

3. **ステップ1-3でメモした`.dump`ファイルを右クリック → 「ダウンロード」**
   - ⚠️ **事故が起きた日時より前のファイル**を選んでください
   - 保存先：`C:\backup\toybox-transfer\`（フォルダを作成してください）
   - **💡 重要**：新しく作成したばかりのファイルが表示されない場合は、以下を試してください：
     - WinSCPの上部メニューから「コマンド」→「更新」をクリック（F5キー）
     - それでも表示されない場合は、WinSCPを一度閉じて再起動してから再接続

4. **右側（サーバー側）で以下のフォルダを開く**：
   ```
   /backup/toybox/volumes/
   ```

5. **ステップ1-3でメモした`media_volume_*.tar.gz`ファイルを右クリック → 「ダウンロード」**
   - ⚠️ **データベースと同じ日時のファイル**を選んでください
   - 保存先：`C:\backup\toybox-transfer\`

6. **WinSCPで復元先サーバーに接続を切り替える**
   - 左上の「新しいサイト」をクリック
   - ホスト名：復元先サーバーのIPアドレス（本番サーバーまたはステージングサーバー）
   - ユーザー名：`root`
   - パスワード：復元先サーバーのパスワード

7. **右側（サーバー側）で以下のフォルダを開く**：
   ```
   /tmp/
   ```

8. **左側（ローカルPC側）で`C:\backup\toybox-transfer\`を開く**

9. **左側のファイルを右側（サーバー側）にドラッグ＆ドロップ**
   - データベースファイル（`.dump`）
   - メディアファイル（`.tar.gz`）

**💡 ヒント**：
- 転送には時間がかかります（メディアファイルは10分以上）
- 画面下部に進行状況バーが表示されます
- ダウンロード・アップロードが途中で止まったら、Cursor AIに相談してください

##### 方法B: サーバー間で直接コピー（より速い・技術者向け）

**復元先サーバー**にSSH接続して以下を実行：

```bash
# PowerShellで復元先サーバーに接続
ssh root@復元先サーバーIP

# 本番サーバーから直接コピー（ファイル名を実際のものに変更）
scp root@本番サーバーIP:/backup/toybox/database/toybox_20260129_153045.dump /tmp/
scp root@本番サーバーIP:/backup/toybox/volumes/media_volume_20260129_153045.tar.gz /tmp/
```

パスワードを聞かれたら、**本番サーバーのパスワード**を入力します。

#### 2-3. 復元先サーバーに接続（まだの場合）

```powershell
ssh root@復元先サーバーIP
```

#### 2-4. データベースをリストア

復元先サーバーで以下を実行：

##### ケースA: `.dump`形式のファイルの場合

```bash
# 1. backendディレクトリに移動
cd /var/www/toybox/backend

# 2. データベースを削除して再作成（UTF-8を明示）
sudo docker compose exec db psql -U postgres -c "DROP DATABASE IF EXISTS toybox;"
sudo docker compose exec db psql -U postgres -c "CREATE DATABASE toybox WITH ENCODING='UTF8' LC_COLLATE='C' LC_CTYPE='C' TEMPLATE=template0;"

# 3. バックアップファイルをコンテナにコピー（ファイル名を実際のものに変更）
# ⚠️ コンテナ名は backend-db-1（db と 1 の間にハイフン。backend-db1 は誤り）
sudo docker cp /tmp/toybox_20260129_153045.dump backend-db-1:/tmp/restore.dump

# 4. リストア実行
sudo docker compose exec db pg_restore -U postgres -d toybox -v /tmp/restore.dump

# 5. マイグレーション実行
sudo docker compose exec web python manage.py migrate

# 6. Webサーバー再起動
sudo docker compose restart web
```

##### ケースB: `.sql.gz`形式のファイルの場合

```bash
# 1. backendディレクトリに移動
cd /var/www/toybox/backend

# 2. データベースを削除して再作成（UTF-8を明示）
sudo docker compose exec db psql -U postgres -c "DROP DATABASE IF EXISTS toybox;"
sudo docker compose exec db psql -U postgres -c "CREATE DATABASE toybox WITH ENCODING='UTF8' LC_COLLATE='C' LC_CTYPE='C' TEMPLATE=template0;"

# 3. 解凍しながらリストア（UTF-8設定を明示・ファイル名を実際のものに変更）
gunzip -c /tmp/toybox_20260129_153045.sql.gz | sudo docker exec \
  -e PGCLIENTENCODING=UTF8 \
  -e LANG=C.UTF-8 \
  -i backend-db-1 psql -U postgres toybox

# 4. マイグレーション実行
sudo docker compose exec web python manage.py migrate

# 5. Webサーバー再起動
sudo docker compose restart web
```

**💡 ポイント**：
- `.dump`形式：`pg_restore`コマンドを使用
- `.sql.gz`形式：`psql`コマンドを使用（UTF-8設定が重要）
- どちらの場合も、データベース作成時に`ENCODING='UTF8'`を明示

**注意**：
- 途中で「already exists」などのエラーが出ても問題ありません
- 最後に `sudo docker compose restart web` を忘れずに！
- **⚠️ ファイル名は必ず実際のバックアップファイル名に変更してください**

#### 2-5. メディアボリュームをリストア

```bash
# 1. 既存のメディアボリュームをクリア
sudo docker run --rm -v backend_media_volume:/data alpine rm -rf /data/*

# 2. バックアップファイルからリストア（ファイル名を実際のものに変更）
sudo docker run --rm \
  -v backend_media_volume:/target \
  -v /tmp/media_volume_20260129_153045.tar.gz:/backup.tar.gz:ro \
  alpine tar xzf /backup.tar.gz -C /target

# 3. 確認
sudo docker run --rm -v backend_media_volume:/data alpine ls -la /data/
```

**⚠️ 重要**：ファイル名は必ず実際のバックアップファイル名に変更してください

#### 2-6. 動作確認

1. ブラウザで復元先サーバーのURLを開く
2. ログインできるか確認
3. 画像が表示されるか確認
4. ゲーム投稿のサムネイルが表示されるか確認

**✅ リストア完了です！**

## パターン2: サーバー→ローカルでの検証

> **例：既存のバックアップを使って、データを自分のPCで確認**

### ステップ1: 既存のバックアップファイルをダウンロード

#### 1-1. ローカルPCに保存先フォルダを作成

1. **エクスプローラーを開く**
2. **C:\**ドライブを開く
3. **右クリック → 新規作成 → フォルダー**で以下を作成：
   - `C:\backup\toybox-verification\database\`
   - `C:\backup\toybox-verification\volumes\`

#### 1-2. WinSCPで本番サーバーに接続

1. **WinSCPを起動**
2. **「新しいサイト」をクリック**
3. **以下を入力**：
   - 転送プロトコル：`SFTP`
   - ホスト名：本番サーバーのIPアドレス（例：`123.456.789.0`）
   - ポート番号：`22`
   - ユーザー名：`root`
   - パスワード：サーバーのパスワード
4. **「ログイン」をクリック**

**💡 ヒント**：
- 初回接続時に「サーバーを信頼しますか？」と聞かれたら「はい」をクリック
- 接続できない場合は、Cursor AIに「WinSCPで接続できません」と相談してください

#### 1-3. データベースファイルをダウンロード

1. **右側（サーバー側）のウィンドウで**フォルダを開く：
   ```
   /backup/toybox/database/
   ```

2. **事故前の適切な日付の`.dump`ファイルを探す**
   - ⚠️ **事故が起きた日時より前のファイル**を選んでください
   - 例：`toybox_20260129_153045.dump`
   - ファイル名の数字が「日付と時刻」を表しています
   - 例：`20260129_153045` = 2026年1月29日 15:30:45
   - **💡 重要**：新しく作成したばかりのファイルが表示されない場合：
     - WinSCPの上部メニューから「コマンド」→「更新」をクリック（またはF5キー）
     - それでも表示されない場合は、WinSCPを一度閉じて再起動してから再接続

3. **ファイルを右クリック → 「ダウンロード」**

4. **ダウンロード先を選択**：
   ```
   C:\backup\toybox-verification\database\
   ```

5. **「OK」をクリック**

**ダウンロード時間**：数秒〜数分

**💡 ヒント**：
- どのファイルを選べばいいか分からない場合は、Cursor AIに相談してください
- 「○月○日の△時頃に問題が起きた」とわかっていれば、それより前のファイルを選びます

#### 1-4. メディアファイルをダウンロード

1. **右側（サーバー側）のウィンドウで**フォルダを開く：
   ```
   /backup/toybox/volumes/
   ```

2. **データベースと同じ日時の`media_volume_*.tar.gz`ファイルを探す**
   - 例：`media_volume_20260129_153045.tar.gz`
   - ⚠️ **データベースと同じ日時のファイル**を選ぶのが重要です
   - **💡 重要**：ファイルが表示されない場合：
     - WinSCPの上部メニューから「コマンド」→「更新」をクリック（またはF5キー）
     - それでも表示されない場合は、WinSCPを一度閉じて再起動してから再接続

3. **ファイルを右クリック → 「ダウンロード」**

4. **ダウンロード先を選択**：
   ```
   C:\backup\toybox-verification\volumes\
   ```

5. **「OK」をクリック**

**ダウンロード時間**：**10分〜30分**（ファイルサイズが大きいため）

**💡 ヒント**：
- ダウンロード中は画面下部に進行状況バーが表示されます
- 転送速度と残り時間が表示されます
- ダウンロードが途中で止まったら、Cursor AIに相談してください

#### 1-5. ダウンロード完了の確認

1. **エクスプローラー**で以下を開く：
   - `C:\backup\toybox-verification\database\`
   - `C:\backup\toybox-verification\volumes\`

2. **ファイルがあることを確認**：
   - データベースファイル（`.dump`）

1. **Windowsの検索で「Docker Desktop」と入力**

```powershell
# 2. バックアップファイルをコンテナにコピー
# ⚠️ ファイル名を実際のファイル名に変更してください！
# ⚠️ コンテナ名は backend-db-1（ハイフンあり）
```

**マイグレーション結果の見方**：

- `Your models in app(s): ... have changes` → リストアの場合は無視してOK。アプリは動作します

**注意**：途中で「could not translate host name "db"」エラーが出た場合は、トラブルシューティングの「could not translate host name "db"」を参照してください。

#### ケースB: `.sql.gz`形式のファイルの場合

```powershell
# 1. データベースを削除して再作成（UTF-8を明示）
docker compose exec db psql -U postgres -c "DROP DATABASE IF EXISTS toybox;"
docker compose exec db psql -U postgres -c "CREATE DATABASE toybox WITH ENCODING='UTF8' LC_COLLATE='C' LC_CTYPE='C' TEMPLATE=template0;"

# 2. ファイルを解凍（7-Zipが必要）
$gzFile = "C:\backup\toybox-verification\database\toybox_20260129_153045.sql.gz"
$sqlFile = $gzFile -replace "\.gz$", ""
& "C:\Program Files\7-Zip\7z.exe" e $gzFile "-o$(Split-Path $sqlFile)" -y

# 3. SQLファイルがUTF-8か確認（日本語が正しく読めるか）
Get-Content $sqlFile -Encoding UTF8 | Select-String "トラブル" | Select-Object -First 1

# 4. SQLファイルをコンテナにコピー
docker cp $sqlFile backend-db-1:/tmp/restore.sql

# 5. コンテナ内で直接リストア実行（文字化け防止・重要！）
docker exec backend-db-1 bash -c "PGCLIENTENCODING=UTF8 psql -U postgres -d toybox -f /tmp/restore.sql"

# 6. 一時ファイルを削除
docker exec backend-db-1 rm /tmp/restore.sql

# 7. マイグレーション実行
docker compose exec web python manage.py migrate

# 8. Webサーバー再起動
docker compose restart web
```

**💡 `.sql.gz`形式の重要ポイント**：
- ❌ **PowerShellから直接パイプでpsqlに流すと文字化けします**
- ✅ **必ずコンテナ内にファイルをコピーしてから、コンテナ内でpsqlを実行**
- `PGCLIENTENCODING=UTF8`の設定が必須

**途中でエラーが出ても大丈夫**：
- 「already exists」→ 既に存在するので無視してOK
- 「permission denied」→ PowerShellを管理者として再起動
- 文字化けが発生したら→ [トラブルシューティング](#問題1-日本語がと表示される)へ

### ステップ5: データベースURLをローカル環境用に変換

**⚠️ 重要**：本番環境のデータをローカルで確認する場合、データベース内のURLは本番サーバーのURLのままです。これをローカル環境用に変換する必要があります。


```powershell
# 3. ユーザーのヘッダー画像を変換

# 4. カード画像を変換（もしあれば）
```

**実行結果の例**：

### 問題1: 日本語が「?」と表示される

リストア後にカード名やユーザー名が「?????????????」と表示される場合の対処法です。

#### 症状例

```
正常：トラブルリキッド・セイバー
文字化け：?????????????
```

#### 原因

データベースに保存される時点で文字化けしています。主な原因：
1. リストア時にUTF-8設定が不足
2. PowerShellパイプ経由でpsqlを実行した（`.sql.gz`の場合）
3. データベース作成時にUTF-8を指定していない

#### 診断手順

```powershell
# データベース内のバイト列を確認
docker exec backend-db-1 psql -U postgres -d toybox -c "SELECT id, encode(name::bytea, 'hex') as name_hex FROM cards WHERE id = 14;"
```

**結果の判定**：
- `3f3f3f3f...`（すべて`3f`）→ **データベースに保存時に文字化け**（以下の対処法Aへ）
- UTF-8の16進数（例：`e38388`で始まる）→ データベース内は正常、表示の問題（対処法Bへ）

#### 対処法A: データベースに保存時に文字化け（最も多いケース）

**解決方法：正しい手順で再リストア**

##### `.dump`形式の場合

```powershell
# 1. データベースをクリア
docker compose exec db psql -U postgres -c "DROP DATABASE IF EXISTS toybox;"
docker compose exec db psql -U postgres -c "CREATE DATABASE toybox WITH ENCODING='UTF8' LC_COLLATE='C' LC_CTYPE='C' TEMPLATE=template0;"

# 2. 再リストア
docker cp "C:\backup\toybox-verification\database\toybox_20260129_153045.dump" backend-db-1:/tmp/restore.dump
docker compose exec db pg_restore -U postgres -d toybox -v /tmp/restore.dump

# 3. 確認
docker exec backend-db-1 psql -U postgres -d toybox -c "SELECT id, name FROM cards WHERE id = 14;"
```

##### `.sql.gz`形式の場合

**⚠️ 重要：コンテナ内で直接実行する必要があります**

```powershell
# 1. データベースをクリア
docker compose exec db psql -U postgres -c "DROP DATABASE IF EXISTS toybox;"
docker compose exec db psql -U postgres -c "CREATE DATABASE toybox WITH ENCODING='UTF8' LC_COLLATE='C' LC_CTYPE='C' TEMPLATE=template0;"

# 2. SQLファイルを解凍（既に解凍済みの場合はスキップ）
$gzFile = "C:\backup\toybox-verification\database\toybox_20260129_153045.sql.gz"
$sqlFile = $gzFile -replace "\.gz$", ""
& "C:\Program Files\7-Zip\7z.exe" e $gzFile "-o$(Split-Path $sqlFile)" -y

# 3. ✅ 正しい方法：コンテナ内にコピーしてから実行
docker cp $sqlFile backend-db-1:/tmp/restore.sql
docker exec backend-db-1 bash -c "PGCLIENTENCODING=UTF8 psql -U postgres -d toybox -f /tmp/restore.sql"
docker exec backend-db-1 rm /tmp/restore.sql

# 4. 確認
docker exec backend-db-1 psql -U postgres -d toybox -c "SELECT id, name FROM cards WHERE id = 14;"
```

**❌ やってはいけない方法**（文字化けします）：

```powershell
# これは文字化けする！
Get-Content $sqlFile -Encoding UTF8 | docker compose exec -T db psql -U postgres toybox
```

**💡 なぜコンテナ内で実行する必要があるのか**：
- PowerShellのパイプ経由だと、文字エンコーディングが変換されてしまう
- コンテナ内で直接実行すれば、UTF-8のまま処理される

#### 対処法B: SQLファイル自体が文字化け

**診断**：

```powershell
# SQLファイルをUTF-8として読んで日本語が正しく表示されるか確認
Get-Content $sqlFile -Encoding UTF8 | Select-String "トラブル"
```

**結果が文字化け**の場合：
- バックアップ元のサーバー側スクリプトを修正が必要
- 新しいバックアップを取得してから再度リストア
- Cursor AIに「バックアップファイル自体が文字化けしています」と相談

### Q1: リストアにどれくらい時間がかかりますか？

**A**: 環境によりますが、目安は以下の通りです：

| 作業 | サーバー間 | サーバー→ローカル |
|------|-----------|------------------|
| バックアップ作成 | 2〜5分 | 2〜5分 |
| ファイル転送 | 1〜3分 | 10〜30分 |
| データベースリストア | 1〜2分 | 1〜2分 |
| メディアリストア | 2〜5分 | 2〜5分 |

### Q2: リストアを途中でやめてしまいました。続きから再開できますか？

**A**: はい、できます！

- データベースリストア途中：ステップ3の最初から再実行
- メディアリストア途中：ステップ4の最初から再実行

**データは上書きされる**ので、途中のデータが残る心配はありません。

### Q3: リストア後、本番サーバーに影響はありますか？

**A**: **影響ありません**。

リストアは、バックアップファイルを**コピーして復元する**作業です。本番サーバーのデータは一切変更されません。

---

### Q4: 複数のバックアップファイルがあります。どれを使えばいいですか？

**A**: **一番新しい日付のファイル**を使ってください。

ファイル名の日付部分を見て判断します：
- ❌ `toybox_20260129_120000.dump` ← 古い
- ✅ `toybox_20260130_153045.dump` ← 新しい（これを使う）

---

### Q5: エラーメッセージが英語で分かりません

**A**: エラーメッセージをコピーして、以下のどれかに該当するか確認してください：

| エラーメッセージ | 意味 | 対処法 |
|-----------------|------|--------|
| `command not found` | コマンドが見つからない | Dockerが起動しているか確認 |
| `permission denied` | 権限がない | PowerShellを管理者として実行 |
| `connection refused` | 接続拒否 | サーバーのIPアドレスを確認 |
| `could not translate host name "db"` | DB接続不可 | `docker compose down` → `up -d` でネットワークリセット |
| `No such container: backend-db1` | コンテナ名のタイプミス | `backend-db-1`（ハイフンあり）に修正 |
| `docker run requires at least 1 argument` | メディアリストアのコマンドが改行された | コマンドを1行で実行 |
| `already exists` | 既に存在する | **無視してOK** |
| `relation does not exist` | テーブルが存在しない | マイグレーションを実行 |

それでも分からない場合は、エラーメッセージ全体をメモして、詳しい人に相談してください。

### Q6: ローカル環境のデータを本番に戻すことはできますか？

**A**: **技術的には可能ですが、強く非推奨です**。

理由：
- 本番データが**完全に上書き**されます
- 他のユーザーのデータが**消えます**
- 取り返しがつきません

本番環境の更新は、必ず以下の手順で行ってください：
1. ステージング環境で十分テスト
2. チーム全員に事前通知
3. バックアップを取ってから実行

## 📚 まとめ

### 文字化けを防ぐための3つの鉄則

| 鉄則 | 内容 |
|------|------|
| **1. データベース作成時** | `CREATE DATABASE` で `ENCODING='UTF8'` を明示的に指定 |
| **2. リストア方法（`.sql.gz`の場合）** | PowerShellパイプは使わず、**コンテナ内で直接psql実行** |
| **3. UTF-8設定** | `PGCLIENTENCODING=UTF8` を設定する |

### コマンド早見表

#### データベースリストア（`.dump`形式）

```bash
# サーバー
# ※ コンテナ名は backend-db-1（ハイフンあり）
sudo docker cp /tmp/backup.dump backend-db-1:/tmp/restore.dump
sudo docker compose exec db pg_restore -U postgres -d toybox -v /tmp/restore.dump

# ローカル
# ※ コンテナ名は backend-db-1（backend-db1 は誤り）
docker cp "C:\backup\backup.dump" backend-db-1:/tmp/restore.dump
docker compose exec db pg_restore -U postgres -d toybox -v /tmp/restore.dump
```

#### データベースリストア（`.sql.gz`形式）

```bash
# サーバー
gunzip -c backup.sql.gz | sudo docker exec -e PGCLIENTENCODING=UTF8 -e LANG=C.UTF-8 -i backend-db-1 psql -U postgres toybox

# ローカル（✅ 正しい方法）
docker cp backup.sql backend-db-1:/tmp/restore.sql
docker exec backend-db-1 bash -c "PGCLIENTENCODING=UTF8 psql -U postgres -d toybox -f /tmp/restore.sql"

# ローカル（❌ 間違った方法・文字化けする）
Get-Content backup.sql | docker exec -T db psql -U postgres toybox
```

#### メディアボリュームリストア

```bash
# サーバー
sudo docker run --rm -v backend_media_volume:/data alpine rm -rf /data/*
sudo docker run --rm -v backend_media_volume:/target -v /tmp/media.tar.gz:/backup.tar.gz:ro alpine tar xzf /backup.tar.gz -C /target

# ローカル
docker run --rm -v backend_media_volume:/data alpine rm -rf /data/*
$BACKUP_FILE = "C:\backup\media.tar.gz"
docker run --rm -v backend_media_volume:/target -v "${BACKUP_FILE}:/backup.tar.gz:ro" alpine tar xzf /backup.tar.gz -C /target
```

### 重要なポイント

1. **事故後に本番サーバーでバックアップを取らない**
   - 既に欠損したデータがバックアップされてしまう
   - 必ず事故前の既存バックアップを使う

2. **データベースとメディアは同じ日時のファイルを使う**
   - 両方が揃って初めて完全なリストアになる
   - 片方だけでは不完全

3. **`.sql.gz`形式の文字化け対策**
   - PowerShellパイプは使わない
   - 必ずコンテナ内にファイルをコピーしてから実行
   - `PGCLIENTENCODING=UTF8`を設定

4. **リストア後は必ず確認**
   - ログインできるか
   - 画像が表示されるか
   - 日本語が「?」になっていないか
