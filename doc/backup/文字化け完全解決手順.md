# 文字化け完全解決手順

**作成日**: 2026年1月29日  
**問題**: リストア後も文字化けが発生する

---

## 問題の原因

リストアしても文字化けが残っている場合、**バックアップファイル自体が既に文字化けしている**可能性があります。

エラー例：
```
ERROR:  could not create unique index "titles_name_key"
DETAIL:  Key (name)=(?????) is duplicated.
```

これは、バックアップ時にUTF-8エンコーディングが設定されていなかったためです。

---

## 解決方法

### ステップ1: サーバー側で新しいバックアップを取得

修正されたバックアップスクリプトを使用して、新しいバックアップを取得します。

#### 1-1. サーバーに接続

```bash
ssh -i "C:\Users\ayato\.ssh\toybox-2025-11-06-11-40.pem" root@160.251.168.144
```

#### 1-2. バックアップスクリプトを実行

```bash
# 修正されたバックアップスクリプトを実行
/var/www/toybox/scripts/backup_database.sh

# ログを確認
tail -50 /var/log/toybox_backup.log

# 作成されたバックアップファイルを確認
ls -lh /backup/toybox/database/ | tail -5
```

#### 1-3. バックアップファイルの内容を確認（文字化けチェック）

```bash
# 最新のバックアップファイルを取得
LATEST_BACKUP=$(ls -t /backup/toybox/database/toybox_*.sql.gz | head -1)

# INSERT文の内容を確認（日本語が正しく表示されるか確認）
gunzip -c "$LATEST_BACKUP" | grep "INSERT INTO.*users" | head -5

# titles テーブルのINSERT文を確認
gunzip -c "$LATEST_BACKUP" | grep "INSERT INTO.*titles" | head -5
```

**期待される結果**: 日本語が正しく表示される（`?????`ではない）

---

### ステップ2: 新しいバックアップをローカルにダウンロード

```powershell
# PowerShellで実行（ローカル環境）
# 最新のバックアップファイル名を確認してから実行
scp -i "C:\Users\ayato\.ssh\toybox-2025-11-06-11-40.pem" root@160.251.168.144:/backup/toybox/database/toybox_$(Get-Date -Format 'yyyyMMdd')_*.sql.gz C:\backup\toybox\database\
```

---

### ステップ3: ローカルでバックアップファイルの内容を確認

```powershell
# 最新のバックアップファイルを取得
$backupFile = Get-ChildItem C:\backup\toybox\database\toybox_*.sql.gz | Sort-Object LastWriteTime -Descending | Select-Object -First 1

# 解凍
$tempSqlFile = $backupFile.FullName -replace '\.gz$', ''
& "C:\Program Files\7-Zip\7z.exe" e $backupFile.FullName "-o$(Split-Path $tempSqlFile)" -y

# 内容を確認（日本語が正しく表示されるか確認）
Get-Content $tempSqlFile -Encoding UTF8 | Select-String "INSERT INTO.*users" | Select-Object -First 5
Get-Content $tempSqlFile -Encoding UTF8 | Select-String "INSERT INTO.*titles" | Select-Object -First 5
```

**期待される結果**: 日本語が正しく表示される

---

### ステップ4: データベースを削除して再作成

```powershell
cd C:\github\toybox\backend
docker compose exec db psql -U postgres -c "DROP DATABASE IF EXISTS toybox;"
docker compose exec db psql -U postgres -c "CREATE DATABASE toybox WITH ENCODING='UTF8' LC_COLLATE='C' LC_CTYPE='C' TEMPLATE=template0;"
```

---

### ステップ5: 新しいバックアップをリストア

```powershell
# UTF-8エンコーディングでリストア
$content = Get-Content $tempSqlFile -Encoding UTF8 -Raw
$content | docker compose exec -e PGCLIENTENCODING=UTF8 -e LANG=C.UTF-8 -T db psql -U postgres toybox

# 一時ファイルを削除
Remove-Item $tempSqlFile
```

---

### ステップ6: リストア結果を確認

```powershell
# 日本語が正しく表示されるか確認
docker compose exec db psql -U postgres -d toybox -c "SELECT id, name FROM gamification_title LIMIT 5;"

# ユーザー情報を確認
docker compose exec db psql -U postgres -d toybox -c "SELECT id, username, display_name FROM users LIMIT 5;"
```

**期待される結果**: 日本語が正しく表示される（`?????`ではない）

## トラブルシューティング

### 問題1: サーバー側のバックアップスクリプトが古い

バックアップスクリプトが修正されていない場合、以下の手順で修正してください：

```bash
# サーバーで実行
# バックアップスクリプトを確認
cat /var/www/toybox/scripts/backup_database.sh | grep "PGCLIENTENCODING"

# 環境変数が設定されていない場合は、スクリプトを修正
# 詳細は「バックアップスクリプト修正手順.md」を参照
```

### 問題2: それでも文字化けが発生する

データベース自体の文字セットを確認してください：

```bash
# サーバーで実行
docker exec backend-db-1 psql -U postgres -d toybox -c "SHOW server_encoding;"
docker exec backend-db-1 psql -U postgres -d toybox -c "SHOW client_encoding;"
```

**期待される結果**: 
- `server_encoding`: `UTF8`
- `client_encoding`: `UTF8`

UTF8でない場合は、データベースを再作成してデータを移行する必要があります。

---

## まとめ

文字化けを完全に解決するには：

1. **サーバー側で新しいバックアップを取得**（UTF-8エンコーディング設定）
2. **新しいバックアップをローカルにダウンロード**
3. **UTF-8エンコーディングでリストア**

古いバックアップファイルが既に文字化けしている場合は、新しいバックアップを取得する必要があります。
