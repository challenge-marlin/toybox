# バックアップがサーバーに保存されない問題 - 診断手順

**作成日**: 2026年2月5日  
**目的**: バックアップがサーバーに保存されない原因を特定し、修正する

---

## 想定される原因（優先度順）

| # | 原因 | 可能性 | 確認方法 |
|---|------|--------|----------|
| 1 | **cron が登録されていない** | 高 | `crontab -l` で確認 |
| 2 | **プロジェクトパスが違う** | 高 | スクリプトは `/var/www/toybox` を想定、deploy は `/home/app/toybox` に配置 |
| 3 | **/backup/toybox ディレクトリが存在しない** | 中 | `ls -ld /backup/toybox` で確認 |
| 4 | **バックアップスクリプトが存在しない** | 中 | プロジェクト内の scripts/ がサーバーにデプロイされているか |
| 5 | **Docker コンテナ名が違う** | 中 | スクリプトは `backend-db-1` を想定 |
| 6 | **ログにエラーが出ている** | - | `/var/log/toybox_backup.log` を確認 |

---

## 診断スクリプト（サーバーで実行）

サーバーに SSH 接続し、以下のコマンドを**root または sudo 権限**で実行してください。

### 一括診断（コピペ用）

```bash
echo "=========================================="
echo "TOYBOX バックアップ診断"
echo "=========================================="
echo ""

echo "【1】cron の登録状況"
echo "---"
crontab -l 2>/dev/null | grep -E "backup|toybox" || echo "❌ バックアップ関連の cron が登録されていません"
echo ""

echo "【2】プロジェクトの配置場所"
echo "---"
for DIR in /var/www/toybox /home/app/toybox /root/toybox; do
  if [ -d "$DIR" ]; then
    echo "✅ 存在: $DIR"
    [ -d "$DIR/scripts" ] && echo "   scripts/: あり" || echo "   scripts/: なし"
    [ -d "$DIR/backend" ] && echo "   backend/: あり" || echo "   backend/: なし"
  fi
done
echo ""

echo "【3】バックアップディレクトリ"
echo "---"
if [ -d /backup/toybox ]; then
  echo "✅ /backup/toybox は存在します"
  ls -ld /backup/toybox/database 2>/dev/null || echo "   database/: なし"
  ls -ld /backup/toybox/volumes 2>/dev/null || echo "   volumes/: なし"
  echo "   既存ファイル数:"
  ls /backup/toybox/database/*.dump 2>/dev/null | wc -l | xargs echo "   - .dump:"
  ls /backup/toybox/database/*.sql.gz 2>/dev/null | wc -l | xargs echo "   - .sql.gz:"
  ls /backup/toybox/volumes/*.tar.gz 2>/dev/null | wc -l | xargs echo "   - volumes .tar.gz:"
else
  echo "❌ /backup/toybox が存在しません"
fi
echo ""

echo "【4】バックアップスクリプトの存在"
echo "---"
for SCRIPT in /var/www/toybox/scripts/backup_nightly.sh /home/app/toybox/scripts/backup_nightly.sh; do
  if [ -f "$SCRIPT" ]; then
    echo "✅ $SCRIPT"
    ls -l "$SCRIPT"
  fi
done
[ ! -f /var/www/toybox/scripts/backup_nightly.sh ] && [ ! -f /home/app/toybox/scripts/backup_nightly.sh ] && echo "❌ backup_nightly.sh が見つかりません"
echo ""

echo "【5】Docker コンテナ"
echo "---"
docker ps --format '{{.Names}}' 2>/dev/null | grep -E "db|backend" || echo "❌ 該当コンテナが見つかりません"
echo ""

echo "【6】バックアップログ（直近20行）"
echo "---"
if [ -f /var/log/toybox_backup.log ]; then
  tail -20 /var/log/toybox_backup.log
else
  echo "❌ /var/log/toybox_backup.log が存在しません（バックアップが一度も実行されていない可能性）"
fi
echo ""
echo "=========================================="
echo "診断完了"
echo "=========================================="
```

## 修正手順

### 修正1: cron を登録する（最重要）

**前提**: プロジェクトが `/home/app/toybox` にある場合と `/var/www/toybox` にある場合でパスが異なります。上記診断で「存在」と表示されたディレクトリのパスを使ってください。

```bash
# プロジェクトが /home/app/toybox の場合
(crontab -l 2>/dev/null; echo "0 21 * * * BACKEND_DIR=/home/app/toybox/backend SCRIPTS_DIR=/home/app/toybox/scripts /home/app/toybox/scripts/backup_nightly.sh") | crontab -

# プロジェクトが /var/www/toybox の場合
(crontab -l 2>/dev/null; echo "0 21 * * * /var/www/toybox/scripts/backup_nightly.sh") | crontab -

# 登録確認
crontab -l
```

**注意**: cron は root ユーザーで登録することを推奨（`sudo crontab -e` または root で `crontab -e`）。`backup_nightly.sh` 内で `sudo` を使用しているため、root の cron で実行するのが確実です。

### 修正2: /backup/toybox ディレクトリを作成

```bash
sudo mkdir -p /backup/toybox/database
sudo mkdir -p /backup/toybox/volumes
sudo chmod 755 /backup/toybox
sudo chmod 755 /backup/toybox/database
sudo chmod 755 /backup/toybox/volumes
ls -ld /backup/toybox /backup/toybox/database /backup/toybox/volumes
```

### 修正3: プロジェクトパスに合わせて backup_nightly.sh のパスを確認

deploy.sh は `/home/app/toybox` にデプロイします。cron で環境変数を渡すか、スクリプト内のデフォルト値を変更してください。

**方法A**: cron で環境変数を指定（推奨）

```bash
# /home/app/toybox にデプロイしている場合
0 21 * * * BACKEND_DIR=/home/app/toybox/backend SCRIPTS_DIR=/home/app/toybox/scripts /home/app/toybox/scripts/backup_nightly.sh
```

**方法B**: シンボリックリンクを作成

```bash
# /var/www/toybox が無い場合
sudo mkdir -p /var/www
sudo ln -sf /home/app/toybox /var/www/toybox
```

### 修正4: 手動でバックアップをテスト

cron 登録前に、手動実行で動作確認してください。

```bash
# プロジェクトが /home/app/toybox の場合
cd /home/app/toybox
sudo BACKEND_DIR=/home/app/toybox/backend SCRIPTS_DIR=/home/app/toybox/scripts ./scripts/backup_nightly.sh

# プロジェクトが /var/www/toybox の場合
sudo /var/www/toybox/scripts/backup_nightly.sh

# 結果確認
ls -lh /backup/toybox/database/
ls -lh /backup/toybox/volumes/
tail -50 /var/log/toybox_backup.log
```

### 修正5: Docker コンテナ名の確認

`backup_database.sh` は `backend-db-1` を想定しています。コンテナ名が異なる場合：

```bash
# 実際のコンテナ名を確認
docker ps --format '{{.Names}}'

# backend ディレクトリで docker compose している場合、通常は backend-db-1
# 異なる場合は backup_database.sh の CONTAINER_NAME を修正
```

## 夜21時に自動バックアップされない原因の特定

「21:00に設定したのに動かない」場合、以下を順に確認してください。

### 確認1: cron が登録されているか

```bash
# root の crontab を確認（バックアップは root で登録する想定）
sudo crontab -l

# または現在のユーザー
crontab -l
```

**結果**: `0 21 * * *` や `backup_nightly` が表示されない → **cron が登録されていません**

### 確認2: cron サービスが動いているか

```bash
systemctl status cron
# または（CentOS/RHEL系）
systemctl status crond
```

**結果**: `inactive (dead)` → サービスを起動: `sudo systemctl start cron`

### 確認3: cron の実行ログ（失敗原因の確認）

```bash
# システムログで cron の実行履歴を確認
grep -i cron /var/log/syslog | tail -30
# または（CentOS/RHEL系）
grep -i crond /var/log/cron | tail -30
```

### 確認4: cron 実行時の典型的な失敗原因

| 原因 | 症状 | 対処 |
|------|------|------|
| **CRLF（改行コード）** | 手動では `bash` で動くが、cron からは失敗 | `sed -i 's/\r$//'` で修正済みか確認 |
| **PATH が空** | cron は PATH が最小限。`docker` や `bash` が見つからない | フルパスで指定（下記参照） |
| **実行ユーザー** | app ユーザーの cron だと `docker` に権限がない | **root の crontab** で登録 |
| **タイムゾーン** | サーバーが UTC の場合、21:00 = 日本時間 6:00（翌日） | `timedatectl` で確認 |

### 正しい cron 登録例（推奨）

```bash
# root で登録（sudo crontab -e または root で crontab -e）
# フルパス使用・bash で実行（CRLF 対策）
(crontab -l 2>/dev/null; echo "0 21 * * * /usr/bin/bash /var/www/toybox/scripts/backup_nightly.sh") | sudo crontab -

# 登録確認
sudo crontab -l
```

### タイムゾーン確認（21:00 が日本時間か）

```bash
timedatectl
# または
date
```

サーバーが UTC の場合、日本時間 21:00 に実行したいなら：

```bash
# 日本時間 21:00 = UTC 12:00
0 12 * * * /usr/bin/bash /var/www/toybox/scripts/backup_nightly.sh
```

### 動作テスト: cron と同じ環境で実行

```bash
# cron はホームが /root で、カレントディレクトリも /root になる
# 環境変数も最小限なので、以下で再現テスト
env -i HOME=/root PATH=/usr/bin:/bin /usr/bin/bash /var/www/toybox/scripts/backup_nightly.sh
```
